### 需求 1：粉丝数量的增长趋势

#### 1.1 用户需求理解

用户希望了解博主粉丝数量在一段时间内的变化情况，判断博主是处于上升期、稳定期还是下降期。这是评估博主潜力的重要指标。

#### 1.2 数据满足情况

| 状态                 | 说明                             |
| -------------------- | -------------------------------- |
| ✅**完全满足** | API 提供完整的 30 天粉丝趋势数据 |

**可用数据来源**: `kol_fans_trend` API

**可用字段**:

```json
{
  "fansNumInc": 112,           // 30天粉丝增长数
  "fansNumIncRate": 0.0725,    // 30天粉丝增长率 (7.25%)
  "list": [                    // 30天每日粉丝数
    {"num": 1544, "dateKey": "2025-11-09"},
    {"num": 1549, "dateKey": "2025-11-10"},
    // ... 共30天数据
    {"num": 1656, "dateKey": "2025-12-08"}
  ]
}
```

**补充数据**（来自 `kol_fans_summary`）:

```json
{
  "fansGrowthRate": "7.3",           // 粉丝增长率
  "fansGrowthBeyondRate": "92.6"     // 增长率超越 92.6% 的同行
}
```

#### 1.3 数据处理方案

1. 直接使用 `fansNumIncRate` 作为 30 天增长率指标
2. 使用 `list` 数据绘制粉丝增长趋势图
3. 可计算增长趋势的斜率判断增长稳定性
4. 使用 `fansGrowthBeyondRate` 与同行对比

### 需求 2：发帖频率

#### 2.1 用户需求理解

用户希望了解博主的内容更新频率，要求"每周 > 3 篇以上"。高频更新说明博主活跃度高，粉丝粘性可能更好。

#### 2.2 数据满足情况

| 状态                   | 说明                                                         |
| ---------------------- | ------------------------------------------------------------ |
| ⚠️**部分满足** | 有笔记列表和活跃度数据，但"过去一个月每周发帖数"需要自行计算 |

**可用数据来源**:

- `kol_note_list` API
- `kol_data_summary_v1` API

**可用字段**:

```json
// kol_note_list
{
  "list": [
    {"date": "2025-12-08", "title": "冬日宅家", ...},
    {"date": "2025-12-06", "title": "宜家居", ...},
    // ... 笔记列表包含发布日期
  ],
  "total": 57  // 总笔记数
}

// kol_data_summary_v1
{
  "noteNumber": 11,           // 近期笔记数(30天内)
  "activeDayInLast7": 7,      // 近7天活跃天数
  "isActive": true            // 是否活跃
}
```

#### 2.3 数据处理方案

1. 筛选 `kol_note_list.list` 中"过去一个月"的笔记（按 `date` 字段过滤）
2. 按周分组统计每周发帖数量
3. 判断是否满足"每周 > 3 篇"的要求
4. 可结合 `activeDayInLast7` 快速判断近期活跃度

#### 2.4 缺失数据

- ❌ 没有直接的"每周发帖数"统计字段
- ❌ 需要自行从笔记列表中计算

### 需求 4：粉丝数据 vs 阅读/评论/互动数据（非投流情况下）

#### 4.1 用户需求理解

用户的筛选标准：

- 单篇阅读数据 > 粉丝数据的 30%
- 单篇评论数据 > 20 条
- 前提条件：非投流（非广告笔记）

这是判断博主"真实流量"和"粉丝活跃度"的关键指标。

#### 4.2 数据满足情况

| 状态                   | 说明                                 |
| ---------------------- | ------------------------------------ |
| ⚠️**部分满足** | 阅读 vs 粉丝可计算，但单篇评论数缺失 |

**✅ 可满足"阅读 vs 粉丝"条件**:

```json
// kol_info
{
  "fansCount": 1656  // 粉丝数
}

// kol_note_list 每篇笔记
{
  "readNum": 3662,           // 单篇阅读数 ✅
  "isAdvertise": false       // 是否广告笔记 ✅ (用于筛选非投流)
}
```

**❌ 无法满足"评论 > 20 条"条件**:

```json
// kol_note_list 每篇笔记包含的字段
{
  "readNum": 3662,      // ✅ 有
  "likeNum": 178,       // ✅ 有
  "collectNum": 113,    // ✅ 有
  // ❌ cmtNum (评论数) 没有！
}
```

#### 4.3 数据处理方案

1. **阅读 vs 粉丝比例**:

   - 从 `kol_note_list` 筛选 `isAdvertise = false` 的笔记
   - 计算每篇笔记的 `readNum / fansCount`
   - 判断是否 > 30%
2. **评论数判断**: **无法实现**

### 需求 5：数据波动性（月内不止一篇爆文）

#### 5.1 用户需求理解

用户希望避免"一篇爆文拉高平均数"的情况，要求博主月内有多篇爆文，说明数据表现稳定而非偶然。

#### 5.2 数据满足情况

| 状态                   | 说明                                               |
| ---------------------- | -------------------------------------------------- |
| ⚠️**部分满足** | 有爆文相关指标，但"爆文"定义不明确，需要自定义标准 |

**可用数据**:

```json
// kol_note_rate
{
  "hundredLikePercent": "9.1",    // 100赞以上笔记占比
  "thousandLikePercent": "0.0",   // 1000赞以上笔记占比
  "notes": [
    {
      "impBeyondRate": 7.888,       // 曝光超越同行比例
      "readBeyondRate": 10.267,     // 阅读超越同行比例
      "interactionBeyondRate": 7.196, // 互动超越同行比例
      // ... 每篇笔记的数据表现
    }
  ]
}
```

#### 5.3 数据处理方案

需要先**定义"爆文"标准**，可能的方案：

- 方案 A：点赞 > 100 即为爆文（使用 `hundredLikePercent`）
- 方案 B：阅读超越同行 > 200%（使用 `readBeyondRate > 2.0`）
- 方案 C：互动数据超过中位数的 3 倍

然后统计"过去一个月"符合条件的笔记数量。

#### 5.4 需要确认

- ❓ **"爆文"的具体定义是什么？** 点赞标准？阅读标准？互动标准？
- ❓ **"不止一篇"是指 ≥2 篇还是 ≥3 篇？**

## 四、补充接口方案

### 4.1 问题背景

蒲公英 API (`kol_note_list`) 返回的笔记列表**缺少评论数**字段：

```json
// kol_note_list 返回 (蒲公英 API)
{
  "readNum": 3662,      // ✅ 有
  "likeNum": 178,       // ✅ 有
  "collectNum": 113,    // ✅ 有
  // ❌ cmtNum 没有！
}
```

### 4.2 解决方案

使用小红书原生接口 `/api/xiaohongshu/get-user-note-list/v4` 获取完整数据：

```json
// user-note-list/v4 返回 (小红书原生 API)
{
  "likes": 3818,           // ✅ 点赞数
  "collected_count": 4375, // ✅ 收藏数
  "comments_count": 203,   // ✅ 评论数 (关键字段！)
  "share_count": 683       // ✅ 分享数 (额外字段)
}
```

### 4.3 接口对比

| 对比项   | 蒲公英 `kol_note_list` | 原生 `user-note-list/v4` |
| -------- | ------------------------ | -------------------------- |
| 参数     | `kolId`                | `userId` (与 kolId 相同) |
| 点赞数   | ✅`likeNum`            | ✅`likes`                |
| 收藏数   | ✅`collectNum`         | ✅`collected_count`      |
| 评论数   | ❌**缺失**         | ✅`comments_count`       |
| 分享数   | ❌ 无                    | ✅`share_count`          |
| 阅读数   | ✅`readNum`            | ❌ 无                      |
| 是否广告 | ✅`isAdvertise`        | ❌ 无                      |
| 翻页方式 | `page` 参数            | `lastCursor` 游标        |

### 4.4 推荐使用策略

**组合使用两个接口获取完整数据**:

2. **原生 API** (`user-note-list/v4`): 获取评论数、分享数

可通过 `noteId` / `id` 字段进行数据关联。
